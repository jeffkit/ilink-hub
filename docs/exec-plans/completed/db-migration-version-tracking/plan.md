# 数据库迁移版本追踪执行计划

目标：修复 ilink-hub 数据库迁移的双轨维护问题 — 添加 `schema_version` 版本追踪、同步 `migrations/` SQL 文件与代码 DDL、消除 ALTER TABLE 静默失败、统一时间戳为 `CURRENT_TIMESTAMP`。

## 里程碑列表

### M1 — schema_version 表与版本追踪原语

在 `run_migrations` 启动时创建 `schema_version` 表，实现原子化 check-and-claim 机制，关闭 TOCTOU 竞态窗口。

- [x] `schema_version` 表（`version INTEGER PRIMARY KEY, migrated_at TEXT DEFAULT CURRENT_TIMESTAMP`）
- [x] `get_current_version()` — 查询当前最高版本（排除并发哨兵行 `i32::MAX`）
- [x] `is_migration_run(version)` — 检查特定版本是否已运行
- [x] `try_claim_migration(version)` — 原子化 claim（SQLite/Postgres: `ON CONFLICT DO NOTHING RETURNING`；MySQL: `INSERT IGNORE`）
- [x] `record_migration_run(version)` — 幂等记录版本
- [x] `try_claim_migration_in_tx(tx, version)` — 事务内原子 claim（F-M3-02 修复）

**验证命令：**
```bash
cargo test --lib store_tests::test_schema_version_tracking -- --nocapture
cargo test --lib store_tests::test_migration_idempotency -- --nocapture
cargo test --lib store_tests::test_migration_incremental_from_v2 -- --nocapture
cargo test --lib store_tests::adversarial_concurrent_store_connect_succeeds_and_converges -- --nocapture
cargo test --lib store_tests::adversarial_many_concurrent_connects_converge -- --nocapture
```

### M2 — 将 run_migrations 拆分为 per-version migrate_to_vN

消除内联大段 DDL，每个迁移版本对应一个独立函数，通过 `schema_version` 做幂等跳过。

- [x] `migrate_to_v1_tx` — 初始 schema（clients, routing_state, context_token_map, bot_credentials）
- [x] `migrate_to_v2_tx` — 后端 session 表（backend_sessions_v2, active_sessions）
- [x] `migrate_to_v3_tx` — `real_ctx` 唯一索引
- [x] `migrate_to_v4_tx` — `context_token_map` 添加 `created_at` 列 + 索引（含列存在预检查）
- [x] `migrate_to_v5_tx` — `messages` 表 + 索引（驱动感知的 `id` 列语法：SQLite `AUTOINCREMENT` / Postgres `GENERATED BY DEFAULT AS IDENTITY` / MySQL `BIGINT AUTO_INCREMENT`）
- [x] 非事务性变体 `migrate_to_vN`（仅测试用，`#[allow(dead_code)]`）
- [x] `run_migrations` 在单个事务中编排 v1-v5，DDL 失败通过 `map_err` + `anyhow!` 向上传播

**验证命令：**
```bash
cargo test --lib store_tests::migration_runs_on_in_memory_sqlite -- --nocapture
cargo test --lib store_tests::test_migration_incremental_from_v2 -- --nocapture
```

### M3 — 同步 migrations/ SQL 文件与内联 DDL

确保 `migrations/*.sql` 文件与 `src/store/mod.rs` 中实际执行的 DDL 完全一致。

- [x] `migrations/0000_schema_version.sql` — 与 `run_migrations` 中的 `CREATE TABLE schema_version` 对齐
- [x] `migrations/0001_initial_schema.sql` — 与 `migrate_to_v1_tx` 对齐，`datetime('now')` → `CURRENT_TIMESTAMP`
- [x] `migrations/0002_backend_sessions.sql` — 与 `migrate_to_v2_tx` 对齐，`datetime('now')` → `CURRENT_TIMESTAMP`
- [x] `migrations/0003_context_token_real_ctx_index.sql` — 与 `migrate_to_v3_tx` 对齐
- [x] `migrations/0004_context_token_map_created_at.sql` — 与 `migrate_to_v4_tx` 对齐，`datetime('now')` → `CURRENT_TIMESTAMP`
- [x] `migrations/0005_messages.sql` — 与 `migrate_to_v5_tx` 对齐（SQLite 形式作为参考）
- [x] `column_exists` 改为 inline tx 查询，避免 `max_connections(1)` 下通过 `self.pool.acquire()` 导致的死锁（F-M3-04）
- [x] `get_current_version` 排除哨兵行（`i32::MAX`），外部调用者看到真实最高版本（F-M3-sentinel）

**验证命令：**
```bash
# 手动审查：对比 migrations/ SQL 文件与代码 DDL
diff <(grep -E 'CREATE|ALTER' src/store/mod.rs) <(grep -E 'CREATE|ALTER' migrations/*.sql)

cargo test --lib store_tests::adversarial_v4_skips_alter_when_column_already_present -- --nocapture
```

### M4 — Review 发现修复

处理 M1/M3 review 中发现的关键问题。

- [x] **F-M1-01 / F-M1-04**：`run_migrations` 改为单连接持有整个迁移过程，关闭跨版本竞态
- [x] **F-M1-02**：v4 的 `ALTER TABLE ADD COLUMN` 使用 catalog 预检查替代错误字符串匹配
- [x] **F-M3-01**：`DatabaseKind` 从 URL scheme 解析，替代运行时的 `SELECT current_database()` 探测（SQLite + MySQL 都会返回 Err，导致 MySQL 被误判为 SQLite）
- [x] **F-M3-02**：并发 `Store::connect` 测试改为 `tokio::join!` 真正并发执行（非顺序 await），验证 `try_claim_migration` 原子性
- [x] **F-M3-04**：`column_exists` 在事务内执行 inline 查询，避免 `max_connections(1)` 下通过 `self.pool.acquire()` 导致的死锁

**验证命令：**
```bash
cargo test --lib store_tests::adversarial_concurrent_store_connect_succeeds_and_converges -- --nocapture
cargo test --lib store_tests::adversarial_many_concurrent_connects_converge -- --nocapture
cargo test --lib store_tests::adversarial_v4_skips_alter_when_column_already_present -- --nocapture
```

### M5 — 深度 Review 修复与版本发布

处理 deep review 发现的问题，准备 v0.2.0。

- [x] **B-01 (DB 超时)**：保持 SQLite 默认 `busy_timeout` 5s，关机时迁移 runner 的 `acquire` 仍有机会排空
- [x] **SEC-002**：`is_safe_identifier` 校验 `pragma_table_info` 拼接，防止 SQL 注入
- [x] **SEC-011**：`Store::connect` 确保在创建池之前调用 `sqlx::any::install_default_drivers()`
- [x] **TO-03**：`tokio::task::spawn_blocking` 调用带超时保护

**验证命令：**
```bash

cargo test --lib
cargo clippy -- -D warnings
```
