## 修复的核心可靠性/安全问题

- **SYNC-02** 活跃路由 vtoken 不同步导致消息路由失败问题 — 在 `upsert_client` 中采用事务，先查询旧 vtoken 再行 upsert。若 vtoken 发生改变，则在同一事务中同步执行 `UPDATE routing_state SET active_vtoken = $new WHERE active_vtoken = $old`，确保活跃路由的 vtoken 状态时刻最新，防止消息被静默丢弃。
- **DB-03** 多数据库元组 `IN` 查询兼容性问题 — 原生的 `(vctx, vtoken) IN (...)` 元组匹配在不同数据库（如 SQLite 或 MySQL 5.7）中支持并不完美。使用 chained `OR` 表达式重构了批量查询的 SQL 构建机制，保证了跨数据库引擎的完全兼容性。
- **DB-02** 大批量广播写锁长占导致阻塞问题 — 在 `persist_context_tokens_batch` 批量持久化时，将原本在单个大事务中写入的所有 tokens 切分为 50 个一组 of chunks。每个 chunk 自主提交事务，极大缩短了单个事务的锁持有时间，防止数据库被持续写入锁死。
- **src/runtime/** — `serve.rs` (+7 / -2) (广播通道容量调优至 1024，并在加载路由时读取 registry 校验 vtoken 存在性)
- **src/hub/** — `mod.rs` (+4 / -1) (在 Metrics 结构体中引入 `dispatcher_lagged` 计数器，并在 spawn_dispatcher lagged 错误时进行累加)
- **src/server/** — `routes.rs` (+8 / -0) (在 `/metrics` 接口输出中注入 `ilink_hub_dispatcher_lagged_total` 指标)
- **tests/** — `hub_routing_integration.rs` (+1 / -2) (清理 unused import `MessageQueue` 以保证 quality-gate 绿灯)

**合计:** 4 文件, +23 / -3 行 (不含 exec-plan / flowx 配置文件)
