## 毒舌架构师 review 关键结论（2026-07-06，基于代码实证）
读过的核心文件：hub/mod.rs, hub/dispatch.rs, hub/queue.rs, hub/router.rs, hub/commands.rs, hub/state.rs, ilink/upstream.rs。
工程完成度高于平均（锁顺序文档+facade、trait 可测、mutation testing、poison-safe mutex、vtoken 日志脱敏都到位）。但有几处用「静默丢数据」换热路径快：
1. **吞吐悬崖**：单 dispatcher 串行消费（dispatch.rs:19-33），quote 路由三层 fallback 每层最多 2s timeout（L1/L2/L3 合计最坏 ~8s），单条慢 DB 查询会序列化全部入站消息。
2. **价值主张裂缝**：context_token 映射持久化是 fire-and-forget + Semaphore(32) try_acquire 失败即 drop（dispatch.rs:172-204, state.rs:485）。核心卖点「映射跨重启持久化」在 DB 抖动/突发时静默丢失，只有计数器无人消费。
3. **单上游 SPOF**：单个 poll loop（upstream.rs:303），-14 re-login 失败 sleep 30s 期间整条管道挂起。by-design 但需文档化 degraded 态。
4. **不一致热点**：ForwardTo 每消息打 DB 取 hub_ext（get_hub_ext_single），broadcast 才批处理（get_hub_ext_batch, dispatch.rs:277）。
5. **send 失败被吞**：upstream.rs:249-252 上游 body 解析失败当 success，只 warn → 用户消息可能静默丢失。
6. **router 独占锁**：state.rs:434 `router: Mutex<Router>`，route() 读多写少却走独占 async mutex。
7. **待核实**：RoutingState 注释「Pure in-memory; no I/O」(state.rs:433)，/use 有 set_route 落库(commands.rs:161)，但 HubState::new 没见 reload——需确认 serve 启动从 DB 重载路由，否则持久化是半成品。
优先级：P0=#2/#5（数据丢失/吞错）；P1=#1/#4/#6（吞吐锁）；P2=#3/#7。
